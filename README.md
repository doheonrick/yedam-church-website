## 1. 프로젝트 목적

예닮교회 웹사이트를 새롭게 리빌딩하여, 방문자와 성도가 필요한 정보를 더 쉽고 빠르게 확인할 수 있는 웹사이트를 만드는 것이 이 프로젝트의 목적이다.

이 프로젝트는 단순히 디자인만 바꾸는 작업이 아니다. 아래 요소들을 함께 개선하는 것을 목표로 한다.

- 더 현대적이고 깔끔한 UI 제공
- 정보 구조 개선
- 모바일 사용성 향상
- 유지보수하기 쉬운 코드베이스 구축
- 향후 백엔드 및 관리자 기능 연동이 가능한 구조 준비

즉, 이 프로젝트의 핵심은 보기 좋은 사이트를 만드는 것을 넘어서, 실제로 운영하기 좋고 오래 유지될 수 있는 사이트를 만드는 것이다.

---

## 2. 프로젝트 개요

기존 예닮교회 웹사이트에는 교회 소개, 예배 안내, 설교, 공지사항 등 중요한 정보가 들어 있다. 하지만 전체적인 정보 구조, 탐색 방식, 모바일 대응, 유지보수성 측면에서는 더 나은 방향으로 개선할 수 있다.

이번 리빌딩은 아래 방향으로 진행한다.

- 사이트 전체 구조 재정리
- 자주 찾는 정보를 더 눈에 띄게 배치
- 재사용 가능한 공통 컴포넌트 설계
- 프론트엔드를 먼저 구현한 후 필요 시 백엔드 연동
- 팀 단위 협업이 가능한 구조 확립

이 프로젝트는 단기적으로는 프론트엔드 웹사이트 구축이 중심이지만, 장기적으로는 관리자 기능 또는 콘텐츠 관리 기능 확장까지 고려할 수 있다.

---

## 3. 프로젝트 목표

이 프로젝트의 주요 목표는 다음과 같다.

- 깔끔하고 현대적인 사용자 인터페이스 구축
- 교회 정보 전달력 향상
- 데스크탑, 태블릿, 모바일 지원
- 사용자 입장에서 이해하기 쉬운 내비게이션 구성
- 프론트엔드를 독립적으로 먼저 개발
- 나중에 실제 데이터 및 API 연동이 쉬운 구조 설계
- 팀 전체가 쉽게 이해하고 수정할 수 있는 코드베이스 구성
- 유지보수성과 확장성을 고려한 구조 설계

## 4. 기술 스택

현재 기준으로 사용하는 기술 스택은 아래와 같다.

### 4.1 프론트엔드

- React
- Vite
- TypeScript
- React Router
- Tailwind CSS

### 4.2 개발 도구

- Git
- GitHub
- VS Code
- ESLint
- Prettier

### 4.3 백엔드

백엔드는 아직 확정되지 않았다.

교회에서 기존 서버나 데이터베이스를 이미 사용하고 있을 가능성이 있으므로, 현재는 프론트엔드를 먼저 개발하고 이후에 백엔드를 연동하는 방향으로 진행한다.

### 4.4 기술 선택 이유

- React: 컴포넌트 기반 구조로 재사용성과 유지보수성이 좋다.
- Vite: 개발 서버가 빠르고 설정이 비교적 간단하다.
- TypeScript: 타입 안정성을 통해 협업과 유지보수에 유리하다.
- React Router: 다중 페이지 구조를 자연스럽게 관리할 수 있다.
- Tailwind CSS: 빠른 UI 구성과 일관된 스타일링에 유리하다.

## 5. 현재 개발 전략

현재 개발 전략은 프론트엔드를 먼저 구현한 뒤, 나중에 백엔드를 연결하는 방식이다.

### 5.1 프론트엔드를 먼저 개발하는 이유

- 교회의 기존 서버 환경이 아직 확인되지 않음
- 페이지 구조와 UI는 백엔드 없이도 먼저 개발 가능함
- 팀원들이 병렬로 작업하기 쉬움
- mock data를 활용하여 초기 개발이 가능함
- 나중에 API를 붙일 때 구조만 잘 잡혀 있으면 교체가 쉬움

### 5.2 현재 단계에서 중요한 원칙

- 데이터와 UI를 최대한 분리한다
- API가 없더라도 서비스 구조를 고려한다
- mock data를 실제 데이터로 쉽게 바꿀 수 있게 작성한다
- 페이지별 책임을 명확히 나눈다
- 하드코딩을 최소화한다

## 8. 팀 개발 원칙

팀이 개발할 때는 아래 원칙을 기본으로 따른다.

### 8.1 명확함 우선

복잡한 구현보다, 누가 봐도 이해하기 쉬운 구조와 코드를 우선한다.

### 8.2 단순한 해결책 먼저

처음부터 너무 큰 구조를 설계하기보다, 현재 필요한 범위에서 단순하고 안정적인 방법으로 구현한다.

### 8.3 과도한 설계 금지

아직 필요하지 않은 기능까지 미리 고려해 구조를 복잡하게 만들지 않는다.

### 8.4 유지보수성 우선

현재 작성하는 코드는 나중에 팀원이 다시 읽고 수정해야 하므로, 가독성과 구조를 중요하게 생각한다.

### 8.5 한 파일, 한 책임

한 파일이 너무 많은 역할을 하지 않도록 한다.

### 8.6 반복이 명확해질 때 공통화

공통 컴포넌트 분리는 실제 반복이 보일 때 진행한다.

좋은 기준은 아래와 같다.

> 먼저 명확하게 동작하게 만들고, 반복이 분명해졌을 때 구조를 개선한다.

## 9. Git 협업 방식

모든 팀원은 아래 Git 협업 방식을 따른다.

- `main` 브랜치에 직접 push 하지 않는다
- 기능 단위 브랜치에서 작업한다
- 작업 완료 후 Pull Request를 생성한다
- 리뷰 후 병합한다
- 하나의 PR에는 하나의 작업 주제만 담는다
- 가능한 한 작은 단위로 작업한다

### 9.1 기본 원칙

1. 안정 브랜치와 작업 브랜치를 분리한다
2. 실수로 운영 코드가 깨지는 상황을 방지한다
3. 리뷰 과정을 통해 코드 품질을 유지한다
4. 변경 이력을 추적하기 쉽게 만든다

## 10. 브랜치 전략

브랜치는 아래 구조를 기준으로 운영한다.

- `main`: 배포 가능한 안정 브랜치
- `dev`: 개발 통합 브랜치
- `feat/...`: 기능 개발 브랜치
- `fix/...`: 버그 수정 브랜치
- `docs/...`: 문서 작성 브랜치
- `refactor/...`: 리팩토링 브랜치
- `chore/...`: 설정 및 기타 유지보수 브랜치

### 10.1 기본 규칙

- `main`에는 직접 push 하지 않는다
- 대부분의 작업 브랜치는 `dev`에서 분기한다
- 완료된 기능은 먼저 `dev`에 병합한다
- 충분히 검토되고 테스트된 후 `main`에 반영한다

### 10.2 브랜치 예시

```bash
main
dev
feat/home-page
feat/sermons-page
fix/mobile-navbar
docs/readme-update
refactor/layout-structure
chore/project-setup
```

## 11. 브랜치 이름 규칙

브랜치 이름은 아래 형식을 따른다.

```bash
타입/설명
```

예시:

```bash
feat/home-page
feat/vision-section
feat/sermon-list
fix/mobile-menu-overlap
docs/readme-update
refactor/common-layout
chore/eslint-config
```

### 11.1 규칙

- 이름만 보고 어떤 작업인지 이해할 수 있어야 한다
- 공백은 사용하지 않는다
- 단어는 `-` 로 연결한다
- 너무 모호한 이름은 피한다
- 한 브랜치는 하나의 목적만 가진다

좋지 않은 예시:

```bash
feat/stuff
fix/thing
update/test
```

좋은 예시:

```bash
feat/church-news-page
fix/footer-spacing-mobile
refactor/shared-button-component
docs/contributing-guide
```

## 12. 커밋 메시지 규칙

커밋 메시지는 아래 형식을 기본으로 한다.

```bash
type: 변경 내용 요약
```

예시:

```bash
feat: 홈 페이지 히어로 섹션 추가
fix: 모바일 헤더 메뉴 겹침 수정
docs: README 전체 정리
refactor: 공통 버튼 컴포넌트 분리
chore: eslint 설정 추가
```

### 12.1 자주 사용하는 타입

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `refactor`: 동작 변화 없는 구조 개선
- `chore`: 설정, 패키지, 기타 유지보수
- `style`: 코드 스타일 정리
- `test`: 테스트 추가 또는 수정

### 12.2 규칙

- 너무 길게 쓰지 않는다
- 어떤 변경인지 바로 이해되게 작성한다
- 한 커밋에는 한 가지 주제만 담는다
- 의미 없는 메시지는 사용하지 않는다

좋지 않은 예시:

```bash
update
fix
changes
final
```

## 13. Pull Request 규칙

모든 변경 사항은 가능한 Pull Request를 통해 병합한다.

### 13.1 PR 기본 원칙

- 한 PR에는 하나의 목적만 담는다
- 설명 없이 PR을 올리지 않는다
- 변경 이유를 분명하게 적는다
- 리뷰어가 이해하기 쉽게 작성한다
- 너무 큰 PR은 피한다

### 13.2 PR 설명 템플릿 예시

```md
## 작업 내용
- 홈 페이지 히어로 섹션 추가
- 공통 버튼 스타일 적용
- 반응형 레이아웃 정리

## 작업 이유
- 메인 화면 첫 인상을 개선하기 위해
- 공통 UI 일관성을 맞추기 위해

## 확인 사항
- 데스크탑/모바일에서 정상 확인
- 기존 라우팅 영향 없음
```

### 13.3 PR 제목 예시

- `feat: 홈 페이지 히어로 섹션 구현`
- `fix: 모바일 헤더 메뉴 오버랩 수정`
- `docs: README 전체 문서화`

## 14. 작업 진행 순서

기본적인 작업 진행 순서는 아래와 같다.

1. 작업 범위 확인
2. `dev` 브랜치 최신화
3. 새 작업 브랜치 생성
4. 기능 구현
5. 로컬에서 확인
6. 코드 정리
7. 커밋
8. 원격 저장소에 push

## 16. 환경 변수 및 설정

VITE_API_BASE_URL=

### 16.3 주의 사항

- 실제 API URL

## 17. 프론트엔드 구조

프론트엔드는 아래 방향으로 구조를 설계한다.

- 페이지와 컴포넌트를 분리한다
- 공통 레이아웃을 재사용 가능하게 만든다
- UI와 데이터 처리 로직을 가능한 분리한다
- 나중에 API 연결이 쉽도록 데이터 접근 구조를 분리한다

기본 구조 예시는 다음과 같다.

- `pages`: 실제 라우팅 페이지
- `components`: 재사용 가능한 UI 컴포넌트
- `layouts`: 공통 레이아웃
- `data`: 임시 mock data
- `types`: 타입 정의
- `services`: API 또는 데이터 처리
- `hooks`: 공통 로직
- `utils`: 유틸 함수

## 20. 라우팅 규칙

라우팅은 React Router 기준으로 관리한다.

### 20.1 기본 원칙

- 의미 없는 경로명은 사용하지 않는다
- 404 페이지도 고려한다

### 20.2 예시 경로

- `/about`: 교회 소개

- `/staff`: 섬기는 분들

- `/sermons`: 설교
- `/news`: 공지사항 / 소식
- `/gallery`: 갤러리
- `/contact`: 오시는 길 / 문의
- `/offering`: 온라인 헌금
- `/english`: 영어 회중

### 20.3 상세 페이지 예시

- `/sermons/:id`
- `/news/:id`
- `/gallery/:id`

## 21. 컴포넌트 설계 규칙

컴포넌트는 아래 기준으로 설계한다.

### 21.1 책임 분리

하나의 컴포넌트는 가능한 한 한 가지 역할에 집중한다.

### 21.2 재사용성 고려

반복되는 UI는 공통 컴포넌트로 분리한다.

예시:

- Button
- SectionTitle
- PageBanner
- Card
- Container

### 21.3 너무 이른 공통화 금지

한 번만 사용한 컴포넌트를 무조건 공통화하지 않는다.

### 21.4 Props 명확화

Props 이름은 의미가 분명해야 하고 불필요하게 복잡하지 않아야 한다.

### 21.5 페이지와 UI 분리

페이지는 화면 조합에 집중하고, 세부 UI는 하위 컴포넌트로 나누는 것을 권장한다.

## 22. 상태 관리 규칙

현재 프로젝트에서는 복잡한 전역 상태 관리 도구를 서두르지 않는다.

### 22.1 기본 원칙

- 단순한 상태는 `useState` 사용
- 생명주기 관련 처리는 `useEffect` 사용
- 여러 컴포넌트에서 반복되는 로직은 custom hook 고려
- 전역 상태가 꼭 필요할 때만 추가 도구 도입 검토

### 22.2 현재 권장 방식

초기에는 아래 방식으로 충분하다.

- 페이지 내부 상태: `useState`
- 간단한 공통 상태: props 전달 또는 context 검토
- 서버 데이터: 추후 API 연결 시 확장

### 22.3 주의 사항

- 불필요한 전역 상태 금지
- 한 컴포넌트에 너무 많은 상태 몰아넣지 않기
- 계산으로 얻을 수 있는 값은 굳이 상태로 두지 않기

## 23. 데이터 및 API 규칙

현재는 mock data 중심으로 개발하되, 나중에 실제 API로 교체하기 쉽게 작성한다.

### 23.1 기본 원칙

- UI에서 직접 하드코딩 데이터를 쓰는 것을 최소화한다
- 데이터 구조는 타입으로 먼저 정의한다
- 실제 API가 없더라도 서비스 계층을 나누는 것을 고려한다

### 23.2 좋지 않은 방식 예시

```tsx
const sermons = [
  { title: "설교1", preacher: "담임목사" }
];
```

### 23.3 더 나은 방식 예시

```tsx
import { getSermons } from "../services/sermonService";
```

### 23.4 추천 방식

- `data/` 폴더에 mock data 보관
- `services/` 폴더에서 데이터 접근
- 페이지는 서비스 결과를 사용
- 나중에 내부 구현만 API 호출로 변경

### 23.5 API 연동 시 고려 사항

- 에러 처리
- 로딩 상태
- 빈 데이터 상태
- 응답 타입 명확화
- 네트워크 실패 대응

## 24. TypeScript 규칙

프로젝트는 TypeScript를 사용하므로 타입 작성을 소홀히 하지 않는다.

### 24.1 기본 원칙

- Props 타입 명시
- API 응답 타입 명시
- 타입 이름은 의미 있게 작성

### 24.2 예시

```ts
export interface SermonItem
{
  id: string;
  title: string;
  preacher: string;
  date: string;
  videoUrl?: string;
}
```

### 24.3 주의 사항

- 임시라고 해서 타입을 생략하지 않는다
- 너무 넓은 타입은 피한다
- 선택 속성은 꼭 필요한 경우에만 사용한다

- 네이밍이 명확한가

- 합의된 규칙 기준으로 리뷰

### 29.3 리뷰 받는 사람의 태도

- 방어적으로 반응하지 않기
- 수정 이유 이해하기
- 필요한 경우 질문하기
- 반영 후 다시 확인 요청하기

- 공통 컴포넌트는 미리 합의한다
- 각자 작업 전에 구조를 공유한다
- 페이지별 스타일이 지나치게 달라지지 않도록 조율한다
- 공통 레이아웃 수정은 팀과 상의 후 진행한다

## 33. 향후 백엔드 연동 메모

현재는 프론트엔드 우선 전략이지만, 향후 아래 기능들은 백엔드와 연결될 가능성이 있다.

- 설교 데이터 관리
- 공지사항 등록 / 수정 / 삭제
- 갤러리 업로드
- 행사 일정 관리
- 문의 폼 제출 처리
- 관리자 로그인
- 교역자 / 사역자 정보 관리
- 주보 / 자료 업로드

### 33.1 지금부터 고려할 점

- 데이터 구조를 타입으로 정리해 둘 것
- mock data와 UI를 분리할 것
- 서비스 계층을 두어 교체 가능하게 만들 것
- 페이지가 특정 데이터 소스에 강하게 묶이지 않도록 할 것

### 33.2 백엔드 확정 전까지의 원칙

- 프론트엔드는 독립적으로 완성도 있게 만든다
- API가 들어올 자리를 미리 생각하되, 지금 과하게 복잡하게 만들지는 않는다

## 34. 배포 메모

배포 방식은 추후 확정될 수 있으나, 프론트엔드 기준으로는 아래 사항을 고려한다.

### 34.1 배포 전 확인 사항

- 빌드 오류가 없는가
- 환경 변수 설정이 분리되어 있는가
- 라우팅 문제가 없는가
- 이미지 경로가 깨지지 않는가
- 모바일에서 주요 페이지가 정상 작동하는가

### 34.2 가능한 배포 후보

- Vercel
- Netlify
- 교회 기존 서버 환경

### 34.3 중요 사항

- 실제 운영 URL 변경 가능성 고려
- 외부 링크와 지도 링크 최종 점검
- 운영용 환경 변수 별도 관리
- 배포 후 모바일 실기기 확인

## 35. 헷갈릴 때 확인할 질문

개발 중 방향이 헷갈릴 때는 아래 질문으로 판단한다.

1. 이 구조가 정말 지금 필요한가?
2. 이 코드를 팀원이 바로 이해할 수 있는가?
3. 이 작업이 현재 범위를 벗어나지 않았는가?
4. 나중에 API를 붙이기 쉬운 구조인가?
5. 모바일에서도 문제없이 보일까?
6. 공통화는 지금 하는 것이 맞는가?
7. README와 실제 구조가 일치하는가?
8. 이 변경이 다른 페이지에 영향을 주지 않는가?

이 질문들에 답하면서 방향을 잡으면 불필요한 복잡도를 줄일 수 있다.

## 36. 마무리

예닮교회 웹사이트 리빌딩 프로젝트는 단순한 디자인 변경이 아니라, 정보 구조, 사용성, 유지보수성, 확장성을 함께 개선하는 작업이다.

현재는 프론트엔드를 먼저 개발하는 전략을 사용하지만, 처음부터 구조를 잘 잡아두면 나중에 백엔드와 관리자 기능이 추가되더라도 훨씬 수월하게 확장할 수 있다.

모든 팀원은 아래를 항상 기억해야 한다.

- 명확한 구조
- 일관된 협업 방식
- 유지보수 가능한 코드
- 모바일까지 고려한 UI
- README와 실제 프로젝트 구조의 일치

이 README는 프로젝트 진행 기준 문서이며, 프로젝트 구조나 협업 방식이 바뀌면 반드시 함께 업데이트해야 한다.
